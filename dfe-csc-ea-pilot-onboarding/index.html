Make better use of children's services data, sector led
Data to Insight
Led by local authorities with support from ADCS, DfE, Ofsted and DLUHC(CSCDUG), working to
help local authorities make better use of data in children's services.


Our focus is assisting and enabling local authority children's data professionals; aligned to developing once, use 153x


We maintain shared toolkits*, support collaborative projects and help build local capabilities within authorities.
Where goals align with government, Data to Insight secures funding so that the service|tool is freely available to every local authority.


*Shared toolkits include such as the ChAT, Benchmarking, CiN 903 Validators, Demand Modelling, RIIA Benchmark and the SSD.

Standard Safeguarding Dataset
SSD components, statutory and indicator items
SSD is a small-footprint, standardised schema that can sit within existing LA reporting structure(s).
It brings together data items present in key statutory and voluntary collections, plus items for priority indicators that are being developed.
Items from statutory and voluntary collections
Data items currently returned within existing stat/voluntary collections
  • CiN Census
  • CLA 903
  • Annex A lists
  • RIIA Quarterly
  • ASGLB child and SGO data
  • SEN2, children with EHCP
  • CSC Workforce Census
  • ADCS Safeguarding Pressures
Items for indicators being developed
Data considered as part of DDSF 1a/b projects.
  • Children looked after via kinship care, 1a(i)
  • Children subject to pre proceedings, 1a(ii)
  • s251 financial data, 1a(iii)
  • Voice of the child, and families, 1a(iv)
Items designed with partners to support new measures and dashboards beyond existing returns.
Standard Safeguarding Dataset
SSD components, extended and new items
SSD also includes additional data most authorities already hold, and new items developed with stakeholders to support emerging frameworks and story telling.
Extended local data
Additional items which most authorities hold but do not currently submit in statutory returns.
  • Data to support SSD outputs for different time periods including multi year longitudinal analysis
  • Immigration status
  • CiN, CP and CLA visit data
  • Address history
  • CP and CLA review data
  • Data recorded consistently across authorities that can support benchmarking and analysis
New items developed with stakeholders
Designed with local authority colleagues and national partners.
  • Items included in the CSC National Framework dashboard
  • Items linked to the CSCS Outcomes Framework
  • Items authorities need to tell important stories about children and families
These additions are co designed so that SSD supports practical local questions as well as national reporting.
Standard Safeguarding Dataset
SSD reductive views
Standard views over the core SSD schema, designed with colleagues, make it easier to use the dataset for local, statuatory and new purposes. Inter-LA data development/reporting is enabled with near-plug-and-play collaborative efforts.
Standard views for specific purposes
Designed from local inputs

The core SSD was designed from inputs provided by local authority colleagues to fulfil existing returns and reporting needs from the core schema. This means reductive views sit on top of the schema|model that is already grounded in real local use cases.

Deployment journey
EA SSD + API deployment, stages summary
A perspective on initial deployment to integration with existing regular server processes.
Technical stages
Stage 1
Setup | smoke tests
  • Deploy SSD core tables and SSD staging tables for the API
  • Schedule one to one sessions to agree support and progress check ins
  • Run local API pre-flight|smoke tests with fake record (PowerShell) to confirm connectivity and whitelisting


  • D2I feedback loop
Stage 2
Local verification
  • Visual checks that SSD tables are populating as expected, not data fixes, focus on flows
  • Visual checks on SSD staging cohort subset
  • Cohort id checks against recent 903 and CiN data
  • Run local API db tests with batched records in payload


  • D2I feedback loop
Stage 3
First live data
  • Send initial live alpha from SSD staging, with hard id filters if helpful during ongoing early cohort checks
  • Manage running cohort cross-checks
  • Access available to the private dashboard once live data is available - start local data fixes as needed

  • D2I feedback loop
Stage 4
Operational rollout
  • Move from local click to run patterns to server based deployments
  • Configure overnight or scheduled runs[SSD refresh + API sender] for ongoing operation





  • D2I feedback loop
