Data to Insight, sector led
Working with local authorities to make better use of children's services data
Led by local authorities with support from ADCS, DfE, Ofsted and DLUHC, working to help local authorities make better use of data in children's services.
Sector led service

Sector led service for local authority children's data professionals.

Maintain shared toolkits, support collaborative projects and help build local capabilities within authorities. Where goals align with government, Data to Insight secures funding so that the service is freely available to every local authority.

Shared toolkits include ChAT, Benchmarking, CiN 903 Validators, Demand Modelling, RIIA Benchmark and SSD.

Standard Safeguarding Dataset
SSD components, extended and new items
SSD also includes additional data most authorities already hold, and new items developed with stakeholders to support emerging frameworks and story telling.
Extended local data
Additional items which most authorities hold but do not currently submit in statutory returns.
  • Data to support SSD outputs for different time periods including multi year longitudinal analysis
  • Immigration status
  • CiN, CP and CLA visit data
  • Address history
  • CP and CLA review data
  • Data recorded consistently across authorities that can support benchmarking and analysis
New items developed with stakeholders
Designed with local authority colleagues and national partners.
  • Items included in the CSC National Framework dashboard
  • Items linked to the CSCS Outcomes Framework
  • Items authorities need to tell important stories about children and families
These additions are co designed so that SSD supports practical local questions as well as national reporting.
Standard Safeguarding Dataset
SSD reductive views
Standard views over the core SSD schema, designed with colleagues, make it easier to use the dataset for specific purposes.
Standard views for specific purposes

SSD reductive views are sets of standardised outputs that present SSD data in simplified, purpose specific formats, for example for returns, dashboards or analysis.

  • Reduce complex schema into views that analysts can work with quickly
  • Support consistent interpretation of measures across authorities
  • Can be aligned to existing local reporting and tools
Designed from local inputs

The core SSD was designed from inputs provided by local authority colleagues to fulfil existing returns and reporting needs from the core schema.

This means reductive views sit on top of a model that is already grounded in real local use cases.

Deployment journey
SSD deployment and onboarding, Stage 1 to Stage 4
A shared view of how we move from initial deployment to regular server based runs, with indicative roles at each step.
Technical stages
Stage 1
Setup and smoke tests
  • Deploy SSD core tables and SSD staging tables for the API
  • Schedule one to one sessions to agree support and progress check ins
  • Run local API smoke tests with fake data using PowerShell or Python to confirm connectivity
Stage 2
Local verification
  • Visual checks that SSD tables are populating as expected, not data fixes, focus on flows
  • Visual checks on SSD staging cohort subset
  • Cross checks against recent 903 and CiN data
Stage 3
First live data
  • Send an initial live alpha from SSD staging, with hard filters if helpful for early testing
  • Provide access to the private dashboard once live data is available
Stage 4
Operational rollout
  • Move from local click to run patterns to server based deployments
  • Configure overnight or scheduled runs for ongoing operation
